1. PROJECT OVERVIEW
================================================================================

PURPOSE
-------
This project demonstrates a complete DevOps CI/CD pipeline built as part of
the Week 2–5 onboarding sprint on the Payment Data Programme team.

The application is a simple log generator. A Bash script (generate_log.sh)
produces an application.log file with timestamped entries. Although the
application itself is simple, the project covers the full enterprise DevOps
toolchain — from writing code, to containerising it, storing the image
in a private registry, automating builds via a CI/CD pipeline, and finally
deploying the container to a Kubernetes cluster.

WHAT THE PROJECT DEMONSTRATES
-------------------------------
  - Source control with Git Flow branching strategy
  - Docker containerisation using a company-approved base image from Nexus3
  - Private image storage in the HSBC Nexus3 registry
  - Automated CI/CD using Jenkins Declarative Pipeline
  - Container orchestration and deployment using Kubernetes
  - Secure credential handling (no passwords in code — Jenkins credentials store)

TARGET AUDIENCE
---------------
This document is written for new joiners who have basic knowledge of Linux
and are beginning to learn DevOps tools. Every step is explained with context
so you understand WHY each action is done, not just WHAT to do.

4.1 PURPOSE
-----------
Git is the version control system used to track all code changes.
GitHub Enterprise (GHE) is the company's internal hosting platform for Git
repositories. It works identically to public GitHub but runs inside the
HSBC network — meaning no external internet access is needed.

Every change made in this project goes through Git before Jenkins picks it up.
Without Git, Jenkins would have nothing to pull and the pipeline could not run.

4.2 BRANCHING STRATEGY — GIT FLOW
-----------------------------------
This project follows Git Flow — the standard branching model used by the team.

  Branch          Purpose
  --------------  -------------------------------------------------------------
  master          Production-ready code only. Protected — no direct pushes.
  develop         Integration branch. All feature work merges here first.
  feature/*       Short-lived branches for individual features or tasks.
  release/*       Pre-release stabilisation branch (used for versioning).

Flow of changes:
  feature/my-task → develop → release/v1.0 → master (tagged v1.0.0)

5. TOOL 2 — DOCKER
================================================================================

5.1 PURPOSE
-----------
Docker packages the application (generate_log.sh) along with everything it
needs (OS, permissions) into a self-contained unit called a container.

Without Docker, the script runs only on the machine where it exists.
With Docker, the script can run identically on any machine — Jenkins server,
developer laptop, or a Kubernetes cluster — because the container carries
its own environment.

Key concept: The container is built from a Dockerfile. The Dockerfile is like
a recipe — it describes exactly how to construct the container image.

IMPORTANT — COMPANY-SPECIFIC BASE IMAGE:
  HSBC does not use public Docker Hub images.
  All base images are pulled from the internal Nexus3 registry.
  Your Dockerfile's FROM line must reference the internal Nexus URL,
  not a public image like "ubuntu:22.04".

  Example format:  FROM nexus3.systems.uk.hsbc:<port>/<base-image>:<tag>
  (Replace with your actual Nexus URL provided by the DevOps team)

5.2 THE FILES
--------------

FILE: generate_log.sh
  This is the application — a Bash script that generates log entries.

  #!/bin/bash
  LOG_FILE="application.log"
  echo "[$(date '+%Y-%m-%d %H:%M:%S')] INFO  - Application started" >> $LOG_FILE
  echo "[$(date '+%Y-%m-%d %H:%M:%S')] INFO  - Processing request" >> $LOG_FILE
  echo "[$(date '+%Y-%m-%d %H:%M:%S')] INFO  - Request completed successfully" >> $LOG_FILE
  echo "[$(date '+%Y-%m-%d %H:%M:%S')] INFO  - Application finished" >> $LOG_FILE
  echo "Log generated: $LOG_FILE"

FILE: Dockerfile
  This file tells Docker how to build the image.

  FROM nexus3.systems.uk.hsbc:<port>/<approved-base-image>:<tag>
  WORKDIR /app
  COPY generate_log.sh .
  RUN chmod +x generate_log.sh
  CMD ["./generate_log.sh"]

  Line-by-line explanation:
    FROM      : Start with the company-approved base OS image from Nexus3
    WORKDIR   : All subsequent commands run inside /app inside the container
    COPY      : Copy the script from our repo into the container at /app/
    RUN       : Make the script executable (chmod +x)
    CMD       : Default command when the container starts — run the script

6. TOOL 3 — NEXUS3 (PRIVATE DOCKER REGISTRY)
================================================================================

6.1 PURPOSE
-----------
Nexus3 is the company's private Docker image registry — it replaces Docker Hub.

Think of it like this:
  - Docker Hub = public supermarket anyone can access
  - Nexus3     = private company warehouse only HSBC systems can access

Two things Nexus3 does in this project:
  1. Hosts the company-approved base images (used in your FROM line)
  2. Stores your built Docker images so Kubernetes can pull them later

Using Nexus3 means no image ever leaves the HSBC network. This is a
security and compliance requirement.

6.2 HOW IMAGES ARE STORED IN NEXUS3
--------------------------------------
After Jenkins builds the Docker image, it tags it with the Nexus3 registry
URL and pushes it. The tag format is:

  nexus3.systems.uk.hsbc:<port>/log-generator:<BUILD_NUMBER>

Every Jenkins build produces a new image tagged with the build number.
Example:
  nexus3.systems.uk.hsbc:8082/log-generator:1
  nexus3.systems.uk.hsbc:8082/log-generator:2
  nexus3.systems.uk.hsbc:8082/log-generator:3
  ... and so on

This gives you a full history of every image ever built — each one traceable
to a specific Jenkins build and Git commit.

6.3 STEP-BY-STEP — NEXUS3 PUSH (DONE BY JENKINS)
----------------------------------------------------
The Nexus3 push is fully automated by Jenkins. Below are the steps that
happen inside the pipeline:

Step 1: Jenkins retrieves Nexus credentials from the credentials store

  No password is hardcoded anywhere. Jenkins uses a "Service Account"
  credential stored securely in Jenkins → Manage Jenkins → Credentials.

Step 2: Jenkins logs in to Nexus3

  In the pipeline (inside withCredentials block):
    docker login nexus3.systems.uk.hsbc:<port> \
      -u "$NEXUS_USER" \
      --password-stdin <<< "$NEXUS_PASS"

Step 3: Jenkins tags the image with the Nexus3 URL

    docker tag log-generator:${BUILD_NUMBER} \
      nexus3.systems.uk.hsbc:<port>/log-generator:${BUILD_NUMBER}

Step 4: Jenkins pushes the tagged image to Nexus3

    docker push nexus3.systems.uk.hsbc:<port>/log-generator:${BUILD_NUMBER}

Step 5: Jenkins logs out (security best practice)

    docker logout nexus3.systems.uk.hsbc:<port>

6.4 VERIFYING THE PUSH IN NEXUS3 UI
--------------------------------------
After the Jenkins pipeline runs successfully:

  1. Open Nexus3 UI in your browser: nexus3.systems.uk.hsbc:<port>
  2. Log in with your credentials

  4. Select your Docker hosted repository

  - Image tagged with Jenkins build number for traceability

7. TOOL 4 — JENKINS (CI/CD PIPELINE)
================================================================================

7.1 PURPOSE
-----------
Jenkins is the automation engine that ties everything together.

Every time you push code to the develop branch on GitHub Enterprise,
Jenkins automatically:
  1. Pulls the latest code
  2. Logs in to Nexus3
  3. Builds the Docker image
  4. Runs the container and generates the log
  5. Archives the log as a build artifact
  6. Pushes the image to Nexus3
  7. Deploys the container to Kubernetes
  8. Cleans up the local Docker image

Without Jenkins, all of these steps would need to be done manually every
time code changes. Jenkins makes the process fast, consistent, and reliable.

7.4 THE JENKINSFILE — COMPLETE PIPELINE
-----------------------------------------
This file lives at: devops_poc/log-generator/Jenkinsfile

--------------------------------------------------------------------------------
pipeline {
    agent any

    environment {
        IMAGE_NAME    = "log-generator"
        IMAGE_TAG     = "${BUILD_NUMBER}"
        NEXUS_REGISTRY = "nexus3.systems.uk.hsbc:<port>"
    }

    stages {

        stage('Checkout') {
            steps {
                echo "Checking out source code..."
                checkout scm
            }
        }

        stage('Docker Login') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'Service Account',
                    usernameVariable: 'NEXUS_USER',
                    passwordVariable: 'NEXUS_PASS'
                )]) {
                    sh """
                        echo "$NEXUS_PASS" | docker login ${NEXUS_REGISTRY} \
                          -u "$NEXUS_USER" --password-stdin
                    """
                }
            }
        }

        stage('Docker Build') {
            steps {
                echo "Building Docker image: ${IMAGE_NAME}:${IMAGE_TAG}"
                sh """
                    docker build \
                      -t ${IMAGE_NAME}:${IMAGE_TAG} \
                      log-generator/
                """
            }
        }

        stage('Docker Run') {
            steps {
                echo "Running container to generate log..."
                sh """
                    docker run --name log-gen-${IMAGE_TAG} \
                      ${IMAGE_NAME}:${IMAGE_TAG}
                    docker cp log-gen-${IMAGE_TAG}:/app/application.log \
                      log-generator/application.log
                    docker rm log-gen-${IMAGE_TAG}
                """
            }
        }

        stage('Archive Log') {
            steps {
                echo "Archiving log file as build artifact..."
                archiveArtifacts artifacts: 'log-generator/application.log',
                                 fingerprint: true
            }
        }

        stage('Push to Nexus') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'Service Account',
                    usernameVariable: 'NEXUS_USER',
                    passwordVariable: 'NEXUS_PASS'
                )]) {
                    sh """
                        docker tag ${IMAGE_NAME}:${IMAGE_TAG} \
                          ${NEXUS_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
                        docker push \
                          ${NEXUS_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
                        docker logout ${NEXUS_REGISTRY}
                    """
                }
            }
        }

        stage('Deploy to Kubernetes') {
            steps {
                withCredentials([file(
                    credentialsId: 'kubeconfig',
                    variable: 'KUBECONFIG'
                )]) {
                    sh """
                        sed -i 's/BUILD_NUMBER_PLACEHOLDER/${IMAGE_TAG}/g' \
                          log-generator/k8s/deployment.yaml
                        kubectl --kubeconfig=$KUBECONFIG apply \
                          -f log-generator/k8s/deployment.yaml
                        kubectl --kubeconfig=$KUBECONFIG rollout status \
                          deployment/log-generator -n devops-poc
                    """
                }
            }
        }
    }

    post {
        success {
            echo "Pipeline succeeded — Build #${BUILD_NUMBER} complete"
        }
        failure {
            echo "Pipeline FAILED — Build #${BUILD_NUMBER}"
        }
        always {
            sh "docker rmi ${IMAGE_NAME}:${IMAGE_TAG} || true"
        }
    }
}
--------------------------------------------------------------------------------

8. TOOL 5 — KUBERNETES
================================================================================

8.1 PURPOSE
-----------
Kubernetes (K8s) is the container orchestration platform — it runs and manages
the Docker containers in production.

Think of Docker as building and running a single container on one machine.
Kubernetes takes that container and:
  - Runs it across multiple machines (nodes)
  - Automatically restarts it if it crashes
  - Updates it to a new version with zero downtime (rolling update)
  - Manages how many copies (replicas) run at any time

In this project, Kubernetes is the final destination for the Docker image.
Jenkins builds the image, pushes it to Nexus3, and then tells Kubernetes
to deploy the latest version.

8.2 KEY KUBERNETES CONCEPTS USED IN THIS PROJECT
--------------------------------------------------

  Concept         What It Is
  --------------  -------------------------------------------------------------
  Deployment      Tells K8s: "Run this container image, keep N copies alive"
  Pod             The running instance of a container
  Namespace       A logical separation/grouping of resources (devops-poc)
  imagePullSecret Credentials K8s uses to pull images from private Nexus3
  Rolling Update  K8s replaces old pods with new ones without downtime
  kubectl         Command-line tool to interact with the K8s cluster

8.3 THE DEPLOYMENT YAML
------------------------
This file lives at: devops_poc/log-generator/k8s/deployment.yaml

The BUILD_NUMBER_PLACEHOLDER text is replaced by Jenkins at deploy time
using the sed command in the pipeline.

--------------------------------------------------------------------------------
apiVersion: apps/v1
kind: Deployment
metadata:
  name: log-generator
  namespace: devops-poc
spec:
  replicas: 1
  selector:
    matchLabels:
      app: log-generator
  template:
    metadata:
      labels:
        app: log-generator
    spec:
      imagePullSecrets:
        - name: nexus-pull-secret
      containers:
        - name: log-generator
          image: nexus3.systems.uk.hsbc:<port>/log-generator:BUILD_NUMBER_PLACEHOLDER
          resources:
            requests:
              memory: "64Mi"
              cpu: "250m"
            limits:
              memory: "128Mi"
              cpu: "500m"
--------------------------------------------------------------------------------

YAML explained:
  replicas: 1         → Run 1 copy of this container
  namespace: devops-poc → Isolate this in the devops-poc namespace
  imagePullSecrets    → Use the nexus-pull-secret to authenticate with Nexus3
  image               → The Docker image URL (BUILD_NUMBER_PLACEHOLDER gets
                        replaced by Jenkins with the actual build number)
  resources           → CPU and memory limits for the container

8.4 ONE-TIME SETUP — imagePullSecret
--------------------------------------
Kubernetes needs a secret to pull images from Nexus3 (private registry).
This is a one-time setup command, run once on the cluster by an admin.

  kubectl create secret docker-registry nexus-pull-secret \
    --docker-server=nexus3.systems.uk.hsbc:<port> \
    --docker-username=<nexus-service-account-user> \
    --docker-password=<nexus-service-account-password> \
    -n devops-poc

  This creates a secret named "nexus-pull-secret" in the devops-poc namespace.
  The deployment.yaml references this secret so K8s can pull the image.

8.5 STEP-BY-STEP — KUBERNETES DEPLOYMENT FLOW
-----------------------------------------------
(Done automatically by Jenkins — documented here for understanding)

Step 1: Jenkins replaces the placeholder in deployment.yaml

  sed -i 's/BUILD_NUMBER_PLACEHOLDER/5/g' log-generator/k8s/deployment.yaml

  Result: The image line becomes:
    nexus3.systems.uk.hsbc:<port>/log-generator:5

Step 2: Jenkins applies the deployment to the cluster

  kubectl --kubeconfig=$KUBECONFIG apply -f log-generator/k8s/deployment.yaml

  What this does:
    - If deployment doesn't exist → K8s creates it
    - If deployment exists → K8s performs a rolling update to the new image

Step 3: Jenkins waits for the rollout to complete

  kubectl --kubeconfig=$KUBECONFIG rollout status \
    deployment/log-generator -n devops-poc

  Output (success):
    "deployment "log-generator" successfully rolled out"

8.6 VERIFYING THE DEPLOYMENT MANUALLY
---------------------------------------
After the pipeline runs, you can verify the deployment using kubectl:

  Check deployment:
    kubectl get deployments -n devops-poc

  Check pods:
    kubectl get pods -n devops-poc

  Describe pod (detailed info):
    kubectl describe pod <pod-name> -n devops-poc

  Check pod logs:
    kubectl logs <pod-name> -n devops-poc

  Screenshot guidance — get pods:
    WHEN  : After a successful Jenkins pipeline run
    WHAT  : Terminal output of "kubectl get pods -n devops-poc" showing
            the log-generator pod with STATUS "Running" or "Completed"
            and RESTARTS as 0

  Screenshot guidance — rollout status:
    WHEN  : In the Jenkins Console Output after the Deploy to Kubernetes stage
    WHAT  : The line in Jenkins console output reading
            "deployment 'log-generator' successfully rolled out"

8.7 EXPECTED OUTPUT / RESULT
------------------------------
  - Deployment exists in the devops-poc namespace
  - Pod shows STATUS: Running (or Completed for a batch/log job)
  - Image tag on the running pod matches the Jenkins build number
  - Rolling update works — old pod replaced by new pod without downtime
  - kubectl rollout status returns "successfully rolled out"

9. END-TO-END PIPELINE FLOW
================================================================================

This section shows the complete journey from a code change to deployment.

TRIGGER: Developer pushes code to develop branch on GitHub Enterprise

  Step 1  → GitHub Enterprise sends webhook to Jenkins
  Step 2  → Jenkins: Checkout — pulls latest code from develop
  Step 3  → Jenkins: Docker Login — authenticates to Nexus3 (Service Account)
  Step 4  → Jenkins: Docker Build — builds image tagged log-generator:7
  Step 5  → Jenkins: Docker Run — container runs, generates application.log
  Step 6  → Jenkins: Archive Log — application.log saved as build artifact
  Step 7  → Jenkins: Push to Nexus — image pushed as nexus3.../log-generator:7
  Step 8  → Jenkins: Deploy to K8s — sed replaces placeholder with :7,
                                     kubectl apply updates the deployment
  Step 9  → Kubernetes pulls nexus3.../log-generator:7 using nexus-pull-secret
  Step 10 → Old pod (log-generator:6) is gracefully stopped
  Step 11 → New pod (log-generator:7) starts and runs
  Step 12 → Jenkins: Cleanup — docker rmi log-generator:7 on Jenkins server
  Step 13 → Jenkins marks build as SUCCESS

RESULT:
  - Confluence-ready artifacts: application.log downloadable from Jenkins
  - Nexus3: new image stored with version :7
  - Kubernetes: pod running version :7
  - GitHub: full commit history in develop branch

VISUAL REPRESENTATION:

  [ GitHub Enterprise ] --webhook--> [ Jenkins ]

                    [ Nexus3 ]                        [ Kubernetes ]
                    (image stored)                  (container deployed)

10. KEY DECISIONS & DESIGN CHOICES
================================================================================

Decision                       Reason
-----------------------------  --------------------------------------------------
Used Nexus3 instead of         HSBC policy — no external Docker Hub access.
Docker Hub                     All images must stay within the HSBC network.

Used Service Account           Security best practice. Passwords are never
credential in Jenkins          written in code. Jenkins credentials store
                               manages them securely.

Used BUILD_NUMBER as           Every build gets a unique, traceable tag.
image tag                      Easy to roll back to any previous version.

Used sed to replace            Keeps deployment.yaml clean in Git.
BUILD_NUMBER_PLACEHOLDER       Jenkins injects the actual build number at
                               deploy time without modifying Git history.

Script Path set to             Allows multiple Jenkins jobs to live in the
log-generator/Jenkinsfile      same repo (monorepo pattern), each with their
                               own Jenkinsfile in their subfolder.

docker rmi in post always      Prevents disk space filling up on the Jenkins
                               server over time. || true ensures cleanup
                               failures do not fail the build.

imagePullSecret in K8s         Required for Kubernetes to authenticate with
deployment.yaml                Nexus3 private registry when pulling images.
