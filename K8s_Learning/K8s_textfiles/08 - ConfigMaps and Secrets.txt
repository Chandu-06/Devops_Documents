1. THE PROBLEM — HARDCODED CONFIGURATION
--------------------------------------------------------------------------------

Imagine you have a banking application that connects to a database.
The database URL is different in each environment:
  - Development:  localhost:5432
  - UAT:          db-uat.hsbc.internal:5432
  - Production:   db-prod.hsbc.internal:5432

You should NEVER hardcode these values into your container image because:
  1. You would need different images for each environment
  2. Changing a config value means rebuilding and redeploying the image
  3. If a database password is in the image, ANYONE with the image has the password
  4. It violates the 12-Factor App principle: separate config from code

GOLDEN RULE:
  Code goes IN containers.
  Configuration goes in ConfigMaps.
  Sensitive data (passwords, tokens, certificates) goes in Secrets.


--------------------------------------------------------------------------------
2. CONFIGMAPS — NON-SENSITIVE CONFIGURATION
--------------------------------------------------------------------------------

A ConfigMap stores non-sensitive configuration data as key-value pairs.
Your application reads these values at runtime through environment variables
or mounted files.


CREATING A CONFIGMAP IN YAML:

    apiVersion: v1
    kind: ConfigMap
    metadata:
      name: payment-api-config
      namespace: hsbc-payments-dev
    data:
      DATABASE_URL: "postgresql://db-service:5432/payments_db"
      LOG_LEVEL: "INFO"
      MAX_CONNECTIONS: "100"
      APP_ENVIRONMENT: "development"
      FEATURE_FLAG_NEW_UI: "false"


USING CONFIGMAP AS ENVIRONMENT VARIABLES (method 1 — all keys at once):

    spec:
      containers:
      - name: payment-api
        image: payment-api:v1.0
        envFrom:
        - configMapRef:
            name: payment-api-config     # Loads ALL keys as environment variables
        # Now DATABASE_URL, LOG_LEVEL, etc. are all available as env vars


USING CONFIGMAP — SPECIFIC KEYS (method 2 — pick individual values):

    spec:
      containers:
      - name: payment-api
        image: payment-api:v1.0
        env:
        - name: DB_URL                  # Name of the env var in the container
          valueFrom:
            configMapKeyRef:
              name: payment-api-config  # ConfigMap name
              key: DATABASE_URL         # Key inside the ConfigMap


USING CONFIGMAP AS A FILE (method 3 — mount as volume):

    spec:
      containers:
      - name: payment-api
        image: payment-api:v1.0
        volumeMounts:
        - name: config-volume
          mountPath: /etc/config        # Files appear here in the container
      volumes:
      - name: config-volume
        configMap:
          name: payment-api-config      # Each key becomes a file in /etc/config/
    # /etc/config/DATABASE_URL contains: postgresql://db-service:5432/payments_db
    # /etc/config/LOG_LEVEL contains: INFO


CONFIGMAP COMMANDS:

    kubectl apply -f configmap.yaml
    kubectl get configmaps
    kubectl get cm                      # 'cm' is the short form
    kubectl describe configmap payment-api-config
    kubectl get configmap payment-api-config -o yaml    # See full content
    kubectl delete configmap payment-api-config

3. SECRETS — SENSITIVE DATA
--------------------------------------------------------------------------------

Secrets are similar to ConfigMaps but designed specifically for sensitive data.
Values are base64-encoded when stored in etcd.

IMPORTANT SECURITY NOTE:
  base64 is NOT encryption. It is encoding. Anyone who can read the Secret
  can easily decode it: echo "c3VwZXJzZWNyZXQ=" | base64 --decode

  Real security comes from:
    1. RBAC - restricting who can read Secrets (see File 14)
    2. Encryption at rest - encrypting etcd data on disk
    3. External secret managers - HashiCorp Vault, AWS Secrets Manager


CREATING A SECRET — ENCODE YOUR VALUES FIRST:

    # Encode values to base64 before putting in YAML
    echo -n "hsbc-db-user" | base64        # outputs: aHNiYy1kYi11c2Vy
    echo -n "P@ssw0rd!2024" | base64       # outputs: UEBzc3cwcmQhMjAyNA==

    apiVersion: v1
    kind: Secret
    metadata:
      name: db-credentials
      namespace: hsbc-payments-dev
    type: Opaque                          # Generic secret type
    data:
      username: aHNiYy1kYi11c2Vy         # base64 of 'hsbc-db-user'
      password: UEBzc3cwcmQhMjAyNA==     # base64 of 'P@ssw0rd!2024'


CREATING A SECRET DIRECTLY FROM COMMAND LINE (simpler — no base64 needed):

    kubectl create secret generic db-credentials \
      --from-literal=username=hsbc-db-user \
      --from-literal=password='P@ssw0rd!2024' \
      -n hsbc-payments-dev
    # kubectl handles the base64 encoding automatically


USING A SECRET AS ENVIRONMENT VARIABLES:

    spec:
      containers:
      - name: payment-api
        image: payment-api:v1.0
        env:
        - name: DB_USERNAME             # Env var name in the container
          valueFrom:
            secretKeyRef:
              name: db-credentials      # Secret name
              key: username             # Key inside the Secret
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password


USING A SECRET AS A MOUNTED FILE (common for TLS certificates):

    spec:
      containers:
      - name: payment-api
        volumeMounts:
        - name: tls-certs
          mountPath: /etc/ssl/certs
          readOnly: true
      volumes:
      - name: tls-certs
        secret:
          secretName: tls-certificate   # Secret containing cert and key files


SECRET TYPES:
  Opaque               — Generic, any key-value data (most common)
  kubernetes.io/tls    — TLS certificates (cert + private key)
  kubernetes.io/dockerconfigjson  — Docker registry credentials (imagePullSecrets)
  kubernetes.io/service-account-token  — Service account tokens (auto-created)


SECRET COMMANDS:

    kubectl create secret generic db-credentials \
      --from-literal=username=admin --from-literal=password=secret
    kubectl get secrets
    kubectl describe secret db-credentials     # Shows type and keys but NOT values
    kubectl get secret db-credentials -o yaml  # Shows base64-encoded values
    kubectl delete secret db-credentials


--------------------------------------------------------------------------------
4. CONFIGMAP vs SECRET — COMPARISON
--------------------------------------------------------------------------------

  Aspect            | ConfigMap                      | Secret
  ------------------|--------------------------------|----------------------------
  Data Type         | Non-sensitive configuration    | Sensitive (passwords, tokens)
  Storage in etcd   | Plain text                     | base64 encoded
  Examples          | DB URL, log level, feature     | DB password, API keys,
                    | flags, app settings            | TLS certificates
  RBAC Access       | Normal access controls         | Stricter — fewer people
  In Pod logs       | May appear in logs             | Should never appear in logs
  Rotation          | Rarely changes                 | Should be rotated regularly

KEY DECISION: If you would be embarrassed if someone saw it, put it in a
Secret. Everything else can go in a ConfigMap.


--------------------------------------------------------------------------------
  REMEMBER: Never commit Secrets (or any sensitive values) to your Git
  repository. Use kubectl create secret from the command line, or use an
  external secrets management tool like HashiCorp Vault.
